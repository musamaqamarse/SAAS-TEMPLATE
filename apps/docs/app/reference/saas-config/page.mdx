# saas.config.json

Written into every scaffolded project root. This is the contract that future lifecycle commands (`update`, `add`, `remove`) read to know what the project actually is.

## Shape

```json
{
  "schemaVersion": 2,
  "createdAt": "2026-05-02T10:00:00.000Z",
  "cliVersion": "0.3.0",
  "project": {
    "name": "My Cool App",
    "kebab": "my-cool-app",
    "snake": "my_cool_app",
    "pascal": "MyCoolApp",
    "bundleId": "com.acme.mycoolapp",
    "description": "A new SaaS project"
  },
  "composition": {
    "dataStack": "supabase",
    "backend": "fastapi",
    "website": "nextjs",
    "adminPanel": "nextjs",
    "mobile": "flutter",
    "includeInfra": true
  },
  "templates": [
    { "name": "backend-fastapi",   "role": "backend",    "version": "1.0.0", "folder": "my-cool-app-backend",    "appliedMigrations": [] },
    { "name": "website-nextjs",    "role": "website",    "version": "1.0.0", "folder": "my-cool-app-website",    "appliedMigrations": [] },
    { "name": "adminpanel-nextjs", "role": "adminpanel", "version": "1.0.0", "folder": "my-cool-app-adminpanel", "appliedMigrations": [] },
    { "name": "mobileapp-flutter", "role": "mobileapp",  "version": "1.0.0", "folder": "my-cool-app-mobileapp",  "appliedMigrations": [] }
  ],
  "agentRulesHash": "sha256:…"
}
```

## Notes

- **Don't edit by hand.** It's a record of how the scaffold was generated. `create-saas update` reads it to know which template version each app started from and computes the migration path forward.
- **Scaffolding-time intent is intentionally absent.** `initGit`, `createGithubRepos`, `githubVisibility`, `destDir` describe the *act* of creation, not the project — they're only on `ScaffoldConfig`, never persisted.
- **`appliedMigrations`** records the IDs of migrations already applied per template, making `update` idempotent.
- **`agentRulesHash`** lets `update`/`add`/`remove` detect whether the user has hand-edited `CLAUDE.md` before deciding to overwrite or write a `.new` sibling.
- **`schemaVersion` is bumped** when the shape changes. v1 files load fine — they're normalized in memory and re-saved as v2 on the first lifecycle command.
